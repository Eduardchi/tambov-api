# Разбор ошибок бэкенда и чек-лист для следующего проекта

Этот файл описывает все проблемы, найденные при первом запуске Laravel-бэкенда
`tambov-api` совместно с фронтендом, причины их появления и как их избежать
в новом проекте с самого начала.

---

## 1. Имя файла Resource не совпадает с именем класса

**Было:** файл `app/Http/Resources/NewsRoesource.php` содержал класс `NewsResource`
(опечатка в имени файла).

**Почему это проблема:** PHP использует PSR-4 автозагрузку — имя файла
**обязано** совпадать с именем класса (с учётом регистра), иначе Composer не
найдёт класс и приложение упадёт с `Class not found`.

**Что делать:**
- При создании классов всегда через `php artisan make:resource ...` —
  Laravel сам создаст файл с правильным именем.
- Если переименовываете класс/файл руками — переименовывайте оба сразу
  (`git mv`, чтобы сохранить историю).
- Перед коммитом можно прогнать `composer dump-autoload` — если есть
  несовпадения, Composer предупредит.

## 2. Namespace класса не совпадает с папкой, в которой он лежит

**Было:** `app/Http/Request/ContactRequest.php` (папка `Request`, без "s"),
но в файле было `namespace App\Http\Requests;` (с "s").

**Почему это проблема:** PSR-4 требует, чтобы путь к файлу повторял
namespace. Несовпадение приводит к `Class not found` при использовании класса.

**Что делать:**
- Всегда создавать Form Request'ы через `php artisan make:request ...` —
  команда сама положит файл в правильную папку `app/Http/Requests/`.
- Если папка создаётся руками — название папки должно быть **точной копией**
  последнего сегмента namespace (с учётом регистра и числа).

---

## 3. Мёртвый `app/Exceptions/Handler.php`

**Было:** в проекте остался файл `app/Exceptions/Handler.php` с логикой
обработки исключений — но в Laravel 11+ этот файл больше не подключается
фреймворком автоматически.

**Почему это проблема:** код в нём просто никогда не выполняется. Это не
ломает приложение, но создаёт ложное ощущение, что обработка ошибок настроена,
хотя по факту нет (стандартные Laravel-ошибки в HTML вместо JSON для API).

**Что делать в Laravel 11+:**
- Всю кастомную обработку исключений писать в `bootstrap/app.php`, в блоке
  `->withExceptions(function (Exceptions $exceptions) { ... })`.
- Если `app/Exceptions/Handler.php` не используется — удалить его, чтобы не
  путать будущих разработчиков.
- Для API-проектов сразу настроить:
  ```php
  $exceptions->shouldRenderJsonWhen(fn ($r) => $r->is('api/*'));
  ```
  и кастомные форматы под коды `404`, `422`, `429` и т.п. — см. итоговый
  `bootstrap/app.php` в этом проекте как референс.

## 4. Отсутствовал `config/cors.php`

**Было:** конфиг CORS не существовал вовсе, хотя фронтенд (Vite, другой порт)
обращался к API из браузера.

**Почему это проблема:** без CORS-конфига браузер блокирует все запросы с
фронтенда на другой origin (другой порт = другой origin). Запросы из Postman/
curl при этом работают нормально — поэтому проблема не видна, если тестировать
без браузера.

**Что делать:**
- В любом проекте, где есть отдельный SPA-фронтенд, сразу создавать
  `config/cors.php` с явным списком разрешённых origin'ов через `.env`:
  ```php
  'paths' => ['api/*'],
  'allowed_methods' => ['*'],
  'allowed_origins' => array_filter(explode(',', env('CORS_ALLOWED_ORIGINS', 'http://localhost:5173'))),
  ```
- **Важно:** список origin'ов должен включать ВСЕ варианты, на которых может
  открываться фронтенд в dev-режиме — `localhost` и `127.0.0.1`, и
  все порты, которые может занять Vite, если основной порт занят
  (5173, 5174, ...). Несовпадение хоста (`localhost` vs `127.0.0.1`) или
  порта — самая частая причина "пустой страницы"/"не загружаются данные" при
  первом запуске.
- Если непонятно, какой origin использует фронтенд — открыть DevTools →
  Network → посмотреть заголовок `Origin` в запросе, который падает с CORS
  ошибкой, и добавить именно его в `CORS_ALLOWED_ORIGINS`.

## 5. `composer.json` указывает PHP `^8.3`, но `composer.lock` требует `8.4+`

**Было:** `composer.json` объявлял совместимость с PHP 8.3, но
`composer.lock` уже был зафиксирован на версиях Symfony 8.1, которым нужен
PHP >= 8.4.1. При попытке `composer install` на PHP 8.3 — конфликт версий.

**Почему это проблема:** несоответствие между декларируемой и фактической
минимальной версией PHP. Любой, кто поднимает окружение по `composer.json`,
получит рабочую среду на 8.3 и упадёт на `composer install`.

**Что делать:**
- После каждого `composer update` проверять актуальную минимальную версию PHP
  в `composer.lock` (секция `platform` / зависимости пакетов) и синхронизировать
  её с `"require": {"php": "^X.Y"}` в `composer.json`.
- В Docker-образе для PHP использовать версию, которая реально требуется
  `composer.lock`, а не ту, что написана в `composer.json` "на глаз".
- Можно зафиксировать версию PHP явно в `composer.json` через `"platform"` в
  `config`, чтобы Composer ругался заранее, а не во время установки на CI/у
  другого разработчика.

## 6. `use Throwable;` в `bootstrap/app.php`

**Было:** добавили `use Throwable;` в начало файла.

**Почему это проблема:** `Throwable` — встроенный интерфейс в глобальном
namespace. `use`-импорт для классов глобального namespace не нужен и вызывает
предупреждение Composer (`The use statement with non-compound name 'Throwable'
has no effect`). Не критично, но "шумит" в логах и указывает на невнимательность.

**Что делать:** не импортировать классы/интерфейсы без namespace
(`Throwable`, `Exception`, `stdClass`, `Closure` и т.п.) — они доступны из
любого namespace без `use`.

## 7. Resource возвращал поле, которого нет в схеме и в таблице

**Было:** `NewsResource::toArray()` возвращал `'location' => $this->location`,
хотя в таблице `news` такой колонки нет (значение всегда `null`), и в
OpenAPI-схеме `NewsItem` такого поля не существует.

**Почему это проблема:** лишнее поле в ответе — это либо мёртвый код
(скопировали Resource по аналогии с другим и не почистили), либо признак
расхождения между API-контрактом (`openapi.yaml`) и реализацией. Фронтенд,
который строго типизирован по схеме, такое поле просто игнорирует — но это
создаёт путаницу при дебаге и риск, что в будущем кто-то начнёт полагаться на
несуществующее поле.

**Что делать:**
- Каждый `*Resource::toArray()` сверять с соответствующей схемой в
  `openapi.yaml` — набор полей должен совпадать 1:1.
- Если Resource создавался "копированием" другого Resource — сразу убирать
  все поля, не относящиеся к текущей модели/таблице.
- Хорошая практика: писать тест (Pest/PHPUnit), который проверяет, что ключи
  JSON-ответа ровно совпадают с ожидаемым набором полей из схемы.

## Общий чек-лист для старта нового проекта (бэкенд + SPA-фронтенд)

1. **Версии PHP**: убедиться, что версия PHP в Docker/локальной среде совпадает
   с тем, что реально требует `composer.lock` (не только `composer.json`).
2. **CORS**: настроить `config/cors.php` сразу, с переменной окружения
   `CORS_ALLOWED_ORIGINS`, включающей все хосты/порты dev-фронтенда
   (`localhost` и `127.0.0.1`, плюс fallback-порты Vite).
3. **Exception handling (Laravel 11+)**: всё в `bootstrap/app.php` →
   `withExceptions()`, файл `app/Exceptions/Handler.php` не использовать/не
   создавать.
4. **Генерация классов через artisan**: `make:resource`, `make:request`,
   `make:controller` и т.п. — чтобы не было рассинхрона имени файла/namespace.
5. **Контракт API**: после написания каждого Resource — сверить поля с
   `openapi.yaml`, убрать всё лишнее.
6. **`.env`**: завести `.env.example` с переменными для БД, CORS, ключей и
   т.п. сразу, чтобы окружение поднималось по одному `cp .env.example .env`.
7. **Проверка интеграции через браузер, а не только curl/Postman**: CORS-
   проблемы не видны в curl — нужно открыть фронтенд в браузере (или
   headless-браузер) и проверить, что запросы реально проходят и страница
   рендерится с данными.
